# GitHub運用方針（経営・実験・実装の切り分け）

**作成日：2025/12/14**

📂 [マニュアル一覧に戻る](./README.md)

---

## 基本原則

- **経営判断・会社方針は「会社の資産」として扱う**
- **思考途中・実験・仮説は個人アカウントで管理する**
- **実装やプロジェクト進行はプルリクエスト前提で運用する**

---

## 🏢 会社GitHubに直接入れるもの（確定・公式）

### 対象
- 経営方針・事業方針
- 事業運営ルール
- 融資・資金調達・M&A方針（確定版）
- 社内向けドキュメント、判断ログ
- 確定した事業計画書・資料

### 運用ルール
- ✅ 原則として `main` ブランチへ直接コミット可
- ✅ 修正履歴は経営判断の履歴として残す
- ✅ レビューは必須としない（スピード優先）
- ✅ コミットメッセージは明確に（何を決定したか）

### リポジトリ例
- `docs` - 経営・事業ドキュメント
- `financial-records` - 会計・経理資料管理
- 確定したプロジェクトのリポジトリ

---

## 👤 個人GitHubで管理するもの（実験・検討段階）

### 対象
- 実験的なビジネスアイデア
- 未検証の収益モデル
- ラフな数値シミュレーション
- 思考メモ、下書き、仮説段階の資料
- プロトタイプ・PoC（概念実証）

### 運用ルール
- ✅ 自由に作成・破棄してよい
- ✅ 原則として外部共有はしない
- ✅ 内容が固まったもののみ会社GitHubへ移す
- ✅ プライベートリポジトリ推奨

---

## 🔄 プルリクエスト運用対象（プロジェクト系）

### 対象
- Webサイト・プロダクト開発
- システム構築・IT実装
- 外注・チーム作業が発生するもの
- 顧客向け成果物

### 運用ルール
- ✅ 必ずブランチを切る
- ✅ プルリクエストで差分を管理する
- ✅ 将来のメンバー増加を前提に記録を残す
- ✅ レビューを経てマージ（チーム作業時）

### ブランチ戦略
```
main (本番)
  ↑
develop (開発)
  ↑
feature/機能名 (機能開発)
```

## 🤔 判断基準（迷ったとき）

| 質問 | YES → | NO → |
|------|-------|------|
| 会社の公式判断か？ | 会社GitHub（直接コミット） | ↓ |
| まだ試行錯誤中か？ | 個人GitHub | ↓ |
| 実装・成果物か？ | 会社GitHub（PR運用） | 個人GitHub |

## 🔐 セキュリティ方針

### 会社リポジトリ
- 機密情報は `.gitignore` で除外
- プライベート設定を維持
- アクセス権限は最小限に

### 個人リポジトリ
- 会社の機密情報は入れない
- プライベート設定推奨
- 必要に応じて会社へ移管

## 🚀 実際の運用フロー

### 経営判断を記録する場合
1. `docs` リポジトリをクローン
2. 該当ファイルを編集
3. コミット＆プッシュ（mainへ直接）
```bash
git add .
git commit -m "Add: 2025年3月の資金調達方針を決定"
git push origin main
```

### 実験的なアイデアを試す場合
1. 個人GitHubで新規リポジトリ作成
2. 自由に試行錯誤
3. 内容が固まったら会社リポジトリへ移行

### プロジェクト開発の場合
1. ブランチを作成
```bash
git checkout -b feature/contact-form
```
2. 開発・コミット
3. プルリクエスト作成
4. レビュー・マージ

## 📝 コミットメッセージ規約

### 会社リポジトリ（docs等）
- `Add: 〜を追加`
- `Update: 〜を更新`
- `Fix: 〜を修正`
- `Decision: 〜を決定`

### プロジェクトリポジトリ
より詳細なコミットメッセージを推奨
- `feat: 新機能追加`
- `fix: バグ修正`
- `docs: ドキュメント更新`
- `style: コードスタイル修正`
- `refactor: リファクタリング`
